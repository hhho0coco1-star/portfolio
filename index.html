LocalQuest

지역을 게임처럼 탐험하는 미션 기반 O2O 로컬 발견 플랫폼

📅 기간: 2026.03.09 ~ 2026.04.03 👥 팀 구성: 5명 🔗 GitHub 링크
Java 11 Spring Framework Spring Security MyBatis JWT React JSP Vanilla JS Oracle Maven Kakao Map API OAuth2 Web Push Python OCR

로컬 상권은 여전히 알려지지 않은 경우가 많고, 소비자는 가까운 지역을 탐험할 동기가 부족합니다. LocalQuest는 퀘스트·포인트·배지 시스템을 통해 지역 탐험을 게임처럼 만들어 O2O 연결을 자연스럽게 유도합니다.

트러블슈팅 및 개선 포인트

문제 상황

QR 코드를 스캔해 검증 페이지(/qr/verify)로 이동했을 때 qrAuthKey 값을 읽지 못해 "유효한 QR 코드가 아닙니다" 오류가 발생. 새로 출력한 QR은 정상 동작하지만 이전에 인쇄해 둔 QR은 계속 실패.

↓
원인 분석

초기 구현에서 buildQrVerifyUrl()이 ?qrAuthKey={uuid}로 URL을 생성했다가, 코드 간결화를 위해 ?key={uuid}로 파라미터 이름을 변경함. QrVerify.js의 searchParams.get() 호출부는 그대로 두어 구 파라미터명을 읽지 못하는 불일치 발생.

↓
해결 과정

QrVerify.js에서 두 파라미터를 모두 수용하도록 searchParams.get('key') || searchParams.get('qrAuthKey') 방어적 fallback 처리를 추가. 이후 파라미터 이름 변경 시 양쪽을 동시에 수정하도록 팀 컨벤션에 명시.

↓
결과

구 파라미터(qrAuthKey)로 생성된 기존 QR과 신 파라미터(key)로 생성된 QR 모두 정상 인증. 파라미터 불일치로 인한 인증 실패 0건.

문제 상황

비즈니스 대시보드의 QR 탭을 반복해서 진입·이탈하면 브라우저 메모리 사용량이 점진적으로 증가. 장시간 사용 시 탭이 느려지는 현상 발생.

↓
원인 분석

API에서 받은 QR 이미지 Blob을 URL.createObjectURL(blob)으로 변환해 <img> src에 바인딩하는 방식. createObjectURL()로 생성된 URL은 명시적으로 revokeObjectURL()을 호출하기 전까지 브라우저 메모리에 유지됨. 초기 구현에서 useEffect cleanup이 없어 unmount 시 해제되지 않음.

↓
해결 과정

useRef(qrObjectUrlRef)로 현재 활성 Blob URL을 추적하고, useEffect의 cleanup 함수(return 블록)에서 URL.revokeObjectURL(qrObjectUrlRef.current)를 호출해 명시적으로 해제. 새 이미지 로드 시에도 이전 URL을 먼저 revoke한 뒤 새 URL을 저장.

↓
결과

페이지 이탈 시 Blob URL이 즉시 해제되어 반복 진입·이탈 후에도 메모리 사용량이 일정하게 유지. 브라우저 DevTools Memory 탭에서 누수 패턴 소멸 확인.

회고 및 배운 점

⚙️

기술적 성장

QR 코드 생성·검증 API를 설계·구현하며 도메인 로직과 인프라 코드를 분리하는 감각을 길렀습니다. Blob URL 메모리 누수 트러블슈팅을 통해 브라우저 리소스 생명 주기와 React useEffect cleanup 패턴을 체득했고, 인터셉터 범위 밖 인증 공백을 직접 발견하면서 보안 관점으로 코드를 바라보는 습관이 생겼습니다.

🤝

협업 경험

트러블슈팅 및 개선 포인트

문제 상황

1:1 선택지 설문에서 사용자가 모든 질문에 답변한 후 성향 분석 결과를 도출할 때, 동일 태그가 여러 질문에 걸쳐 중복 등장하는 경우 결과가 부정확하게 산출되는 오류가 발생했습니다. 일부 케이스에서 실제 가장 많이 선택한 태그가 아닌 다른 태그가 최종 성향으로 결정되었습니다.

↓
원인 분석

초기 로직은 '태그 유무(존재 여부)'만 비교하는 방식이었습니다. 각 질문에서 선택한 태그를 Set에 넣어 중복을 제거한 뒤 비교하다 보니, 같은 태그가 여러 질문에서 반복 선택되어도 1회로만 카운트되었습니다. 그 결과 실제 선택 빈도가 낮은 태그가 Set 기준으로 먼저 등장해 잘못된 성향으로 결정되는 경우가 발생했습니다.

↓
해결 과정

'태그 유무 비교' 방식을 폐기하고, '질문별 가중치 합산' 방식으로 로직을 전면 수정했습니다. 각 질문에서 선택한 태그에 질문별 가중치를 부여하고, 동일 태그가 여러 질문에서 선택될수록 합산 점수가 누적되도록 Map<String, Integer> 구조로 변경했습니다. 전체 질문 답변 완료 후 가장 높은 합산 점수를 가진 태그를 최종 여행 스타일로 결정합니다.

↓
결과

중복 태그가 발생하는 모든 케이스에서 사용자가 실제로 가장 많이 선택한 태그가 정확하게 최종 여행 스타일로 결정됩니다. 로직 수정 후 성향 분석 결과 오류 케이스 0건을 확인했습니다.

문제 상황

여행지 상세 정보(영업시간 등)를 가져오는 과정에서 페이지 로딩 속도가 현저히 느려지는 현상이 발생했습니다. 수많은 여행지 데이터를 처리할수록 응답 지연이 심해졌고, 사용자 경험에 직접적인 영향을 주었습니다.

↓
원인 분석

여행지 데이터를 순차적(Sequential)으로 하나씩 API 호출·DB 조회하는 구조였습니다. 데이터 양에 비례하여 전체 응답 시간이 누적되는 구조적 문제였으며, 앞선 처리가 완료될 때까지 다음 처리가 시작되지 못하는 Blocking이 발생했습니다. CPU·네트워크 자원을 충분히 활용할 수 있음에도 한 번에 하나의 작업만 수행하여 전체 처리 효율(Throughput)이 저하되었습니다.

↓
해결 과정

Java의 CompletableFuture 또는 Parallel Stream을 활용하여 단건 호출 방식을 병렬 처리 구조로 변경했습니다. 여러 여행지 데이터를 동시에 처리하도록 설계함으로써 Blocking 없이 자원을 효율적으로 활용할 수 있게 되었습니다.

↓
결과

기존 순차 처리 대비 데이터 로딩 속도를 획기적으로 단축했습니다. 병렬 처리 도입 후 기존 대비 약 70~80%의 성능 향상을 확인했습니다.

기술적 성장
