HashTrip

여행 성향 분석부터 일정 공유까지, 내 취향에 맞는 여행을 계획하는 플랫폼

📅 기간: 2026.02.09 ~ 2026.03.04 👥 팀 구성: 5명 🔗 GitHub 링크
Java 11 Spring Framework Spring Security MyBatis OAuth2 Spring Mail JSP Vanilla JS Oracle Maven

여행 계획을 세울 때 어디를 가야 할지 막막함을 느끼는 경우가 많습니다. HashTrip은 설문 기반 성향 분석으로 개인 맞춤 여행 스타일을 도출하고, 한국관광공사 공공API 기반 여행지 데이터를 활용해 체계적인 일정 계획과 공유를 지원합니다.

🧭 여행 성향 분석 설문 기반 태그 가중치 합산으로 개인 맞춤 여행 스타일 도출
📅 여행 일정 계획 공공API 여행지 기반 일정 생성·공유 및 루트 북마크
🗺️ 여행지 탐색 한국관광공사 API 기반 장소 검색·평점·운영시간 제공
🔑 소셜 로그인 Google · Kakao · Naver OAuth2 통합 인증 지원

핵심 도메인 설계

성향 분석 · 여행 일정 흐름 중심 핵심 테이블 발췌

  1. Place ↔ Tag를 N:M 중간 테이블(Place_Tag_Map)로 분리하고, tag_weight · tag_confidence 컬럼을 두어 AI 자동 분류 이력과 신뢰도를 관리했습니다.
  2. 사용자 성향(User_Tag_Map)을 설문 질문별로 분리 저장해, 질문 단위 가중치 합산 방식의 유연한 성향 분석을 지원했습니다.
  3. Travel_Plans와 Plan_Details를 분리해 계획 단위(공개/비공개)와 장소별 방문 순서(plan_visit_order)를 독립적으로 관리했습니다.

트러블슈팅 및 개선 포인트

문제 상황

1:1 선택지 설문에서 사용자가 모든 질문에 답변한 후 성향 분석 결과를 도출할 때, 동일 태그가 여러 질문에 걸쳐 중복 등장하는 경우 결과가 부정확하게 산출되는 오류가 발생했습니다. 일부 케이스에서 실제 가장 많이 선택한 태그가 아닌 다른 태그가 최종 성향으로 결정되었습니다.

↓
원인 분석

초기 로직은 '태그 유무(존재 여부)'만 비교하는 방식이었습니다. 각 질문에서 선택한 태그를 Set에 넣어 중복을 제거한 뒤 비교하다 보니, 같은 태그가 여러 질문에서 반복 선택되어도 1회로만 카운트되었습니다. 그 결과 실제 선택 빈도가 낮은 태그가 Set 기준으로 먼저 등장해 잘못된 성향으로 결정되는 경우가 발생했습니다.

↓
해결 과정

'태그 유무 비교' 방식을 폐기하고, '질문별 가중치 합산' 방식으로 로직을 전면 수정했습니다. 각 질문에서 선택한 태그에 질문별 가중치를 부여하고, 동일 태그가 여러 질문에서 선택될수록 합산 점수가 누적되도록 Map<String, Integer> 구조로 변경했습니다. 전체 질문 답변 완료 후 가장 높은 합산 점수를 가진 태그를 최종 여행 스타일로 결정합니다.

↓
결과

중복 태그가 발생하는 모든 케이스에서 사용자가 실제로 가장 많이 선택한 태그가 정확하게 최종 여행 스타일로 결정됩니다. 로직 수정 후 성향 분석 결과 오류 케이스 0건을 확인했습니다.

문제 상황

여행지 상세 정보(영업시간 등)를 가져오는 과정에서 페이지 로딩 속도가 현저히 느려지는 현상이 발생했습니다. 수많은 여행지 데이터를 처리할수록 응답 지연이 심해졌고, 사용자 경험에 직접적인 영향을 주었습니다.

↓
원인 분석

여행지 데이터를 순차적(Sequential)으로 하나씩 API 호출·DB 조회하는 구조였습니다. 데이터 양에 비례하여 전체 응답 시간이 누적되는 구조적 문제였으며, 앞선 처리가 완료될 때까지 다음 처리가 시작되지 못하는 Blocking이 발생했습니다.

↓
해결 과정

Java의 CompletableFuture 또는 Parallel Stream을 활용하여 단건 호출 방식을 병렬 처리 구조로 변경했습니다. 여러 여행지 데이터를 동시에 처리하도록 설계함으로써 Blocking 없이 자원을 효율적으로 활용할 수 있게 되었습니다.

↓
결과

기존 순차 처리 대비 데이터 로딩 속도를 획기적으로 단축했습니다. 병렬 처리 도입 후 기존 대비 약 70~80%의 성능 향상을 확인했습니다.

회고 및 배운 점

⚙️

기술적 성장

한국관광공사 공공API 대량 데이터를 ExecutorService 병렬 처리와 배치 INSERT로 효율적으로 수집하는 방법을 익혔습니다. 성향 분석 알고리즘을 직접 설계하고, 태그 중복 처리 버그를 발견·수정하는 과정에서 알고리즘의 엣지 케이스를 꼼꼼히 검토하는 습관이 생겼습니다. Spring Security OAuth2 소셜 로그인 3종 연동을 통해 인증 흐름 전반을 이해하게 되었습니다.

🤝

협업 경험
